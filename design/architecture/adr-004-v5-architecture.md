**日期**: 2026-05-08
**状态**: Proposed

## ADR-004: v5.0 模块化架构（完全重写）

### Context
v4.3.7 是一个 3492 行的单体 game.js，虽然有 4 模块拆分，但系统耦合严重。
v5.0 根据用户完整策划案，需要支持全新系统：
- 6 格悬浮卡槽 + 溢出替换
- 5 格道具栏 + 物理掉落 + 跟随 Boss 道具
- 全局天赋系统（杀敌数驱动）
- 两套英雄体系（无UI精准对子流 / 带UI修仙藏剑 L1-L3 回收）
- 英雄蓝图 HUD（查阅/操作隔离）
- 特权出怪口系统

### Decision
采用 **16 模块分层架构**，全局作用域 + 严格加载顺序：

```
Layer 0 (Engine): engine-core → engine-input → engine-audio → engine-render
Layer 1 (Data):   game-data → game-state
Layer 2 (Logic):  game-loop → game-combat → game-waves → game-cards → game-items → game-talents → game-heroes → game-cultivation
Layer 3 (UI):     game-ui → game-main
```

### Rationale
- 全局作用域：与 v4 保持一致，无需构建工具
- 分层加载：引擎层零游戏逻辑依赖，数据层仅配置，逻辑层全部系统，UI 层最后
- 16 个文件替代 1 个 3492 行文件
- 每个文件职责单一，便于独立开发和调试

### Consequences
- 需要严格管理全局命名空间（window.*），避免冲突
- 加载顺序必须精确，否则依赖未定义
- 调试时需要打开多个文件，但每个文件更小更聚焦

## ADR-005: 相机系统

### Context
v4 的相机系统简单跟随玩家，无平滑过渡。v5 需要：
- 镜头严格锁定人物居中
- 怪物从屏幕上端口涌入的压迫感需要相机在垂直方向有张力
- 修仙体系的跟随 UI 需要知道相机偏移

### Decision
采用 **平滑跟随相机（Lerp Camera）**：
- 目标位置 = 玩家世界坐标
- 实际位置 = lerp(当前位置, 目标位置, 0.1)
- 相机偏移量作为全局变量暴露（cameraX, cameraY）
- 垂直方向增加轻微 overshoot（进入新区域时相机略滞后，产生"拖拽感"）

### Rationale
- lerp 平滑过渡避免突兀跳动
- 垂直滞后增强"怪物从上方涌入"的压迫感
- 全局偏移量方便所有 UI 元素计算屏幕位置

## ADR-006: 双轨经济

### Context
v4 只有金币单一资源。v5 引入双轨经济：金币（卡牌） + 杀敌数（天赋）。

### Decision
- 金币：击杀怪物获得，用于抽卡（6格卡槽）
- 杀敌数：击杀怪物累积，满阈值触发天赋选择
- 两个计数器独立追踪，互不干扰
- 杀敌数阈值按难度缩放（n1: 50杀/天赋, n10: 20杀/天赋）

### Rationale
- 双轨强制玩家在"立即强化（抽卡）"和"长期投资（天赋）"之间做战术取舍
- 不引入第三种资源，保持简洁
